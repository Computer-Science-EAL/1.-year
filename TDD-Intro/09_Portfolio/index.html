StockTest*:

First of all! You should complete this assignement/lab before moving on to the next

Imagine you're writing a stock trading program. The most important feature is to get a list of the user's current stock portfolio, and to calculate its value. So we start off writing an object that encapsulates a single stock, with fields (also known as variables, properties, or members) for symbol, price per share, and number of shares. We also want a method that computes the dollar value per stock (price per share x number of shares).

Use variables of type double for price and value.

Notes on individual tests:

CanSetAndGetSymbol, CanSetAndGetPrice, CanSetAndGetNumShares: write a straightforward property for these three properties.

CanSetAndGetNumShares also requires a calculator method, GetValue, that computes the value of the stock by multiplying the price by the number of shares.

CanUseMultiArgumentConstructor: a constructor is called when an object is first created (instantiated). This is usually a good place to set the initial values of all properties. Note that if you don't specify any constructor, the compiler always adds an implicit default constructor (one that takes no arguments); this means that if you add a non-default constructor, you will have to also add a default constructor if you want the old tests to work. This can be done with a simple 

public Stock() { }

CanCalculateTotalValue: This test calls a static method called TotalValue that takes an array of Stocks as a parameter. A static method is attached to the object class itself, but not to any specific instance of that object. The declaration for the TotalValue method should be something like 

public static double TotalValue(Stock[] stocks)

CanCreateToString: Every C# object has a method called ToString. It returns a string representing that object and its values. ToString is supposed to be used for debugging, not for showing to an end-user, so its output does not have to be pretty. However, the default implementation of ToString is really not pretty, so most people override ToString and write their own. The CanCreateToString method asks you to implement ToString following the standard idiom of printing the name of the class, then its properties surrounded by [ square brackets ].

(When you call System.Console.Write(foo), the Write method actually invokes foo's ToString method and prints that value.)

TwoStockObjectsWithSameValuesAreEqual: C# has two main concepts of equality. The first is, "Are these two object references literally the same instance?" That is embodied by the Equals method. The second is, "Are the contents of these two objects the same?" That is emodied by the == operator. This test assures that two separate objects, created with the same values, are truly counted as equal. There are however ways to test the content of two objects by still using the Equals method

TwoStockObjectsWithDifferentValuesAreNotEqual: This debugs your equals method, making sure that you are comparing all values and so forth.

PolymorphismTest*:

Polymorphism is the object-oriented property whereby two objects of different classes can be treated identically, at least to a certain extent. Polymorphism is implemented in C# in two ways: inheritance and interfaces. To implement these tests, you must first create an interface called Asset, then make your Stock object implement the Asset parent. The only thing an Asset needs to do, for present purposes, is return its value, so Asset will only need a GetValue method.

You will then create another type of asset, a Savings Account. Then you will see how useful polymorphism is -- it allows you to (very slightly) change your TotalValue method so it works on any type of asset, not just stocks.

(From an object-oriented design point of view, this new TotalValue method will be slightly out-of-place. It's still in the Stock class, but is not restricted to Stocks. We are about to clean this up by making a new object, Portfolio, that is devoted to holding collections of assets.)

PortfolioTest*:

Finally, we will make a Portfolio object, responsible for holding a group of assets. Here let's not worry too much about making this object robust -- we will do that later, in the Collections lab. For now, just make sure it will work for a collection of a few assets.
